# 笔记

## 学习笔记：项目结构设计

### 选择的结构

基于学习内容，我为LLM Bootstrapping项目设计了以下结构：

text

```
llm_bootstrap/
├── src/
│   └── llm_bootstrap/
│       ├── __init__.py
│       ├── config.py
│       ├── logger.py
│       └── core.py
├── scripts/
│   └── hello_cli.py
├── configs/
│   └── config.yaml
├── .env
├── pyproject.toml
├── README.md
└── docs/
    └── engineering_mapping.md
```

### 采用此结构的原因（≥200字）

我选择**src布局**而非扁平布局，主要原因有三：

1. ​**避免导入混淆**​：Python优先导入当前目录下的模块。src布局确保`python -m`和测试时导入的是已安装的包版本，而非开发目录中的源码，这能及早发现缺失依赖或导入路径问题。
2. ​**清晰的关注点分离**​：将源代码(`src/`​)、脚本(`scripts/`​)、配置(`configs/`​)、文档(`docs/`)分离，使项目结构更直观，符合"约定优于配置"原则，降低认知负担。
3. ​**符合现代Python打包标准**​：PEP 518和621推荐使用`pyproject.toml`​，结合src布局能更好地与构建工具(如`hatch`​、`pdm`​)和包管理器集成。同时，支持`pip install -e .`进行可编辑安装，便于开发调试。

对于​**入口脚本设计**​，我选择`python -m`​方式而非直接执行`.py`​文件。因为`python -m llm_bootstrap.cli`​能确保模块导入路径正确，且能通过`console_scripts`在安装后提供全局命令。

在**配置管理**上，采用层级策略：环境变量用于密钥等敏感信息(.env文件管理)，YAML配置用于项目参数(结构化配置)，命令行参数用于运行时调整。日志系统采用模块化配置，不同组件可独立控制日志级别。

这种结构虽稍显复杂，但为LLM项目的长期演进奠定了基础——当模型训练、API集成、评估等模块增加时，能保持代码组织清晰。
